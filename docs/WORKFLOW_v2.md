## 서비스 모델 (최종 정리)

```
┌─────────────────────────────────────────────────────────────┐
│  무료 사용자 (로컬)                                           │
│  ─────────────────                                           │
│  • MCP → 로컬 AI(Claude Desktop)에 연결해서 자연어로 검색/주문   │
│  • localhost:5900 웹UI → 브라우저에서 직접 검색/주문             │

│  유료 사용자 (클라우드)                                        │
│  ─────────────────                                           │
│  • 서버가 24/7 자동검색 + 자동주문까지 수행                      │
│  • 팜스퀘어 로그인 → domae.pharmsquare.com 대시보드              │
│  • Basic(30분) / Pro(5분, 3약국)                              │
│  • 핵심 가치: "출근하면 밤새 최저가 주문이 완료되어 있음"          │
└─────────────────────────────────────────────────────────────┘
```

**첫 실행 감지 로직**:
```
App.jsx의 useEffect에서:
1. GET /api/settings/credentials → 모두 미설정이면 setup 필요
2. config에 api_key가 비어있으면 setup 필요
3. setup 필요 → /setup으로 리다이렉트
4. 설정 완료 → localStorage에 'setup_complete' = true 저장
```

**API 키 검증 연동**:
```
SettingsPage의 handleVerify() 완성:
1. PUT /api/settings/api-key { api_key: "dmk_free_xxx" }
2. POST /api/settings/api-key/verify → ApiKeyManager.verify() 호출
3. 응답: { valid, tier, pharmacy_name, features }
4. UI에 결과 표시 ("마이약국 — Free 등급 ✅")
```

**필요한 백엔드 추가**:
```
local/routers/settings.py에 추가:
- PUT /api/settings/api-key        → config에 저장
- POST /api/settings/api-key/verify → ApiKeyManager.verify() 호출, 결과 반환
- GET /api/settings/setup-status   → { api_key_set, credentials_count, telegram_set }
```

### 5B-1. Windows 원클릭 설치 (1일)

```
install.bat:

1. Python 설치 확인 → 없으면 winget으로 자동 설치
2. git clone (또는 zip 다운로드 + 압축해제)
3. pip install -r requirements.txt
4. frontend 빌드 (또는 pre-built static 포함)
5. 바탕화면에 바로가기 생성 ("도매 통합검색.lnk")
6. 최초 실행

→ 사용자는 install.bat만 더블클릭하면 됨
```

**대안** (더 간단): pre-built static을 git에 포함시켜 npm build 단계 제거
```
src/domae_mcp/static/에 빌드된 프론트엔드 커밋
→ 설치: git clone + pip install만으로 끝
→ install.bat에서 node.js/npm 불필요
```

```

### 6A-1. 하트비트 수집 서버

```
팜스퀘어 서버 또는 별도 서비스:

→ 이미 로컬 클라이언트(api_key.py)가 12시간마다 전송 중
→ 받는 쪽만 구현하면 됨
```

### 6A-2. 간단 어드민 대시보드

```
pharmsquare.com/admin/domae (또는 별도)

표시할 것:

- 버전 분포 (1.0.0: 80%, 0.9.0: 20%)
- API 키 목록 (발급일, 마지막 활동, 등급)
```

> 목표: 유료 사용자에게 "24/7 자동검색 + 자동주문" 제공

> 의존성: Phase 5 완료 + 팜스퀘어 JWT 검증 API 확보

### 6B-1. 프로젝트 셋업 + 인프라 (2일)

```
maipharm-domae-cloud/
├── cloud/
│   ├── auth/
│   │   ├── pharmsquare.py    # JWT 검증 (팜스퀘어 공개키로 verify)
│   │   └── dependencies.py   # FastAPI Depends: get_current_user
│   ├── database.py           # PostgreSQL + user_id FK
│   ├── config.py             # 환경변수 기반 (not file-based)
│   └── models/
│       ├── domae_user.py     # user_id, pharmacy_name, tier, api_key
│       └── user_credential.py # AES-256 암호화 (서버 관리 키)

```
docker-compose:
  api:        FastAPI 웹서버

  postgres:   메인 DB
  nginx:      리버스 프록시 (domae.pharmsquare.com)
```

### 6B-2. 멀티테넌트 DB + 인증 (3일)

```
기존 로컬 모델에 user_id 추가:
- Product → user_id FK
- Order → user_id FK
- UrgentOrder → user_id FK
- MonitorSchedule → user_id FK
- InventorySnapshot → user_id FK

인증 흐름:
1. 사용자가 팜스퀘어 로그인 → JWT 쿠키 발급 (팜스퀘어가 해줌)
2. domae.pharmsquare.com 접속 → JWT 쿠키 자동 전송
3. 클라우드 API 미들웨어가 JWT verify → user_id 추출
4. 모든 쿼리에 .filter(user_id=current_user.id) 적용
```

### 6B-3. 클라우드 라우터 (3일)

```
cloud/routers/ — 로컬 코어 서비스를 JWT 인증으로 감싸기

핵심 차이점 (로컬과 다른 부분만):
1. 모든 엔드포인트에 get_current_user 의존성
2. 크리덴셜 저장: AES-256 (서버 관리 키) — 로컬의 Fernet과 다름
3. 스케줄 저장 시 Celery Beat 스케줄 업데이트 트리거
4. 검색/주문 결과를 user별로 격리
```

### 6B-4. Celery 자동화 엔진 (3일) ← 클라우드 핵심 가치

```
cloud/tasks/

monitor_task.py:
  @celery.task
  def run_monitor_cycle(user_id):
    1. user의 모니터링 제품 목록 조회
    2. user의 크리덴셜로 크롤러 인스턴스 생성
    3. 각 제품 검색 → 이전 스냅샷 비교
    4. 가격 변동 / 재고 변동 감지
    5. 알림 전송 (텔레그램 + 카카오)

urgent_order_task.py:
  @celery.task
  def check_urgent_orders(user_id):
    1. 활성 긴급주문 조회
    2. 해당 제품 검색 → 재고 확인
    3. 재고 있으면 → 자동 주문 실행
    4. 결과 로그 저장 + 알림

celery_beat_schedule:
  Basic 사용자: 30분마다 monitor_task + urgent_order_task
  Pro 사용자: 5분마다
  → 요금제 변경 시 동적으로 스케줄 갱신
```

**⚠ 자동주문 안전장치 설계 (중요)**:

```
자동주문 발동 조건:
1. 사용자가 긴급주문으로 "명시적 등록"한 제품만 대상
2. total_quantity 초과 주문 불가
3. 1회 주문량 제한 (max_per_order, 기본 10박스)
4. 일일 자동주문 금액 상한 (daily_limit, 기본 50만원)
5. 주문 성공 시 즉시 카카오 알림 (취소 가능 링크 포함?)

자동주문이 "아닌" 것:
- 모니터링 = 가격/재고 변동 "알림"만 (주문 X)
- 일반 검색 = 사용자가 직접 주문
- 긴급주문 = 사용자가 조건 설정 → 서버가 자동 실행

→ "출근하면 밤새 주문 완료" = 긴급주문 기능의 클라우드 확장
→ 로컬에서도 긴급주문은 있지만, PC 꺼지면 중단
→ 클라우드는 24/7이라 실질적 자동화 달성
```

### 6B-5. 카카오 알림톡 기본 연동 (2일)

```
cloud/notifications/kakao_service.py

카카오 비즈니스 채널 등록 (maipharm):
  - 채널 프로필 설정
  - 템플릿 심사 (최소 3개):
    1. "재고 감지 알림" — #{제품명} 재고 #{수량}개 감지 (#{도매상}, #{가격}원)
    2. "자동 주문 완료" — #{제품명} #{수량}개 주문 완료 (#{도매상}, #{금액}원)
    3. "가격 변동 알림" — #{제품명} 가격 변동: #{이전가}→#{현재가} (#{도매상})

사용자 연결:
  팜스퀘어에서 카카오 인증 → 알림톡 수신 동의
  (팜스퀘어가 이미 카카오 연동 있으면 재활용)
```

### 6B-6. 클라우드 프론트엔드 (3일)
